# Layout Labo Requirements

## 1. 文書方針

本書は Layout Labo の **現在有効な仕様だけを記載する正本** とする。

- 実装判断は本書を唯一の参照源とする
- 仕様変更の経緯、比較案、採否理由は `docs/decisions.md` に記録する
- 同一事項を複数箇所へ重複記載しない
- 仕様変更時は文末追記ではなく、既存の該当節へ統合更新する
- 既存記述と矛盾する記述を残置してはならない

## 2. プラグイン概要

### 2.1 目的

本プラグインは、QGIS レイアウトの地図アイテム情報を扱う統合ツールである。

主な目的は次の通り。

- 既存レイアウトから地図アイテム情報を CSV へエクスポートする
- CSV の情報を既存レイアウトへインポートする
- 新しいレイアウトテンプレートを用いてレイアウトを再作成する
- 地図アイテム間コピー、expression 生成適用、zvalue 再設定を行う

### 2.2 動作環境

- QGIS 3.4 以上
- Python（PyQGIS）

### 2.3 用語

#### 2.3.1 表示内容

本プラグインにおける「表示内容」とは、地図アイテムに設定された次の data-defined property を指す。

`dataDefinedMapLayers.expression`

レイアウト XML では次の位置に存在する。

`LayoutItem > dataDefinedProperties > properties > dataDefinedMapLayers > expression`

例

`<Option type="QString" value="'Google Road'" name="expression"/>`

### 2.4 対象情報

本プラグインが主に扱う情報は次の通り。

- 地図アイテムの Extent
- `dataDefinedMapLayers.expression`

### 2.5 対象外

少なくとも次は本プラグインの対象外とする。

- レイヤ可視状態
- 地図テーマ
- 縮尺
- 回転
- ラベル設定
- その他描画設定

## 3. 共通仕様

### 3.1 提供機能

本プラグインは次の機能を提供する。

- エクスポート
- インポート
- 再作成
- 地図アイテム任意コピー
- expression ビルダ
- ページ考慮 zvalue 再設定

### 3.2 レイアウト解決規則

- インポート時は `layout_name` とプロジェクト内レイアウト名の **完全一致** のみを適用対象とする
- レイアウト名の部分一致、あいまい一致、手動対応付けは行わない
- 再作成時の対象選択は CSV 内レイアウト一覧を用いる
- 再作成時は、プロジェクト内レイアウトの存在有無によって対象候補を制限しない

### 3.3 地図アイテム解決規則

#### 3.3.1 CSV ベース機能

エクスポート、インポート、再作成での地図アイテム対応は `itemById()` により行う。

- 対応の起点は CSV データ側の地図アイテムとする
- ID は必ずしもユニークではない
- `itemById()` が返す最初の一致アイテムを対象とする
- `map_item_id` 空値は想定しない

#### 3.3.2 同一 ID 重複対応機能

地図コピー機能と expression ビルダ機能では、`itemById()` の先頭一致のみで対象を確定しない。

- `map_item_id`
- 同一 `map_item_id` 内の出現順（`occurrence_index`）

の組で地図アイテムを解決する。

### 3.4 警告とエラー

警告が発生した場合は、メインダイアログ内へメッセージ表示する。

ログレベルは次を使い分ける。

- `INFO`
- `WARNING`
- `ERROR`

### 3.5 `_要確認` 規則

- 再作成処理で警告が発生したレイアウトは、レイアウト名末尾に `_要確認` を付与する
- 例: `市街図_要確認`
- インポート、地図コピー、expression ビルダ、zvalue 再設定ではレイアウト名変更を行わない

### 3.6 地図アイテム数不一致

地図アイテム数比較は、各レイアウト内の **全地図アイテム数** で行う。

- テンプレートの地図アイテム数 < CSV データの地図アイテム数 の場合は警告し、移せる地図アイテムのみ反映する
- テンプレートの地図アイテム数 > CSV データの地図アイテム数 の場合は警告し、余った地図アイテムはテンプレート初期値のまま残す
- インポート時に CSV 内レイアウトと既存レイアウトの全地図アイテム数が一致しない場合も警告し、一致した地図アイテムへの適用は継続する

### 3.7 レイアウト一覧と関連状態の再取得

メインダイアログ表示中に扱うレイアウト一覧や関連状態を固定キャッシュしてはならない。

少なくとも次の契機で、必要な一覧や関連状態を再取得する。

- ダイアログ初回表示
- 各タブ表示時
- 明示的な再読込操作時
- 各処理完了後

### 3.8 処理単位

警告は次の単位で保持、表示できること。

- レイアウト単位
- 地図アイテム単位

例

- `レイアウトAで不一致が発生しました`
- `レイアウトA: 地図1 がテンプレートで見つかりません`

### 3.9 成功判定ルール

- エクスポート:
  - CSV 書き出し成功かつ全レイアウト成功の場合 `success=True`
  - CSV 書き出し成功かつ一部レイアウト失敗の場合も `success=True` とし、`WARNING` を付与する
  - CSV 書き出し失敗時は `success=False`
- インポート / 再作成 / zvalue 再設定:
  - 失敗レイアウトが 1 件でもあれば `success=False`
  - 警告のみの場合は `success=True` を維持する
- CSV 読み込み:
  - 不正行が 1 行でもあれば致命扱いで `success=False` とし、データセットを空として返す

## 4. データ仕様

### 4.1 内部データ構造

CSV 入出力と QGIS レイアウト操作の間に、QGIS API オブジェクトへ過度に依存しない内部データ構造を持つこと。

少なくとも次の 3 段階で扱えること。

- CSV 1 行相当
- 1 レイアウト分
- CSV 全体

### 4.2 CSV 1 行相当データ

1 行は少なくとも次を保持する。

- `layout_name`
- `map_item_id`
- `xmin`
- `ymin`
- `xmax`
- `ymax`
- `expression`

この構造は 1 つの地図アイテムに対応する。

### 4.3 地図アイテムデータ

地図アイテムデータは少なくとも次を表現する。

- 所属レイアウト
- 地図アイテム ID
- 適用すべき Extent
- 適用すべき `dataDefinedMapLayers.expression`

Extent は文字列 1 本ではなく、`xmin` `ymin` `xmax` `ymax` の数値として保持する。

### 4.4 レイアウト単位データ

同一 `layout_name` を持つ地図アイテムデータを束ねて、1 レイアウト分のデータとして扱う。

少なくとも次を保持できること。

- `layout_name`
- 地図アイテムデータ一覧

必要に応じて次を保持してよい。

- 警告有無
- 警告メッセージ一覧
- 元 CSV 行番号一覧

### 4.5 CSV 全体データ

CSV ファイル全体は複数レイアウト分のデータを保持する構造として扱う。

少なくとも次の操作を提供できること。

- `layout_name` 一覧の取得
- 特定 `layout_name` の取得
- レイアウト名単位での反復処理

### 4.6 値型

- `xmin` `ymin` `xmax` `ymax` は内部的には数値として扱い、CSV 出力時のみ文字列化する
- `expression` は文字列として保持する
- `expression` の空文字は有効値として扱ってよい
- 不要なトリム、加工、式変換は行わない
- `layout_name` と `map_item_id` は文字列として保持する

### 4.7 CSV 仕様

本プラグインで扱う CSV は、本プラグイン自身が出力し、本プラグイン自身が読み込む固定仕様とする。

一般的な任意 CSV の広範囲な互換性は目的としない。

#### 4.7.1 エンコーディング

CSV の文字コードは BOM 付き UTF-8 とする。

#### 4.7.2 ヘッダー

CSV はヘッダー行を必須とし、次の固定列を持つこと。

- `layout_name`
- `map_item_id`
- `xmin`
- `ymin`
- `xmax`
- `ymax`
- `expression`

#### 4.7.3 列順

CSV の列順は次の通り固定とする。

1. `layout_name`
2. `map_item_id`
3. `xmin`
4. `ymin`
5. `xmax`
6. `ymax`
7. `expression`

書き出し時は必ずこの順序で出力すること。

#### 4.7.4 文字列列

文字列列は次を含む。

- `layout_name`
- `map_item_id`
- `expression`

これらは CSV 出力時に `"` で括り、必要なエスケープは CSV ライブラリの標準挙動へ委ねる。

#### 4.7.5 数値列

数値列は次を含む。

- `xmin`
- `ymin`
- `xmax`
- `ymax`

不要な丸めを行わず、表示用の桁整形も行わない。

#### 4.7.6 expression の保持

`expression` は CSV にそのまま保存する。

次を禁止する。

- 前後空白の自動除去
- クォート記法の書き換え
- 式の再解釈
- 式の正規化
- 独自実装による任意のエスケープ置換

#### 4.7.7 エスケープ対象文字

少なくとも次を含む文字列を扱えること。

- カンマ
- `"`
- 改行

独自の文字列連結ではなく、標準的な CSV ライブラリで入出力すること。

#### 4.7.8 読み込み失敗条件

次の場合は読み込み失敗とする。

- 必須列不足
- `layout_name` 空値
- `map_item_id` 空値
- `xmin` `ymin` `xmax` `ymax` 空値
- 数値変換失敗
- 同一 `layout_name` かつ同一 `map_item_id` の重複行

#### 4.7.9 空ファイル

ヘッダーのみで実データ行が存在しない CSV は読み込み可能としてよいが、適用対象なしとして扱う。

この場合は `WARNING` または `INFO` を出力する。

#### 4.7.10 改行コード

改行コードは実行環境依存でよいが、標準的な CSV ライブラリで互換性を確保する。

#### 4.7.11 書き出しの一貫性

本プラグインが書き出した CSV は、本プラグイン自身でそのまま再読込可能であること。

### 4.8 バリデーション責務

CSV 読み込み時に、内部データ構造へ変換する前または変換時に最低限の妥当性確認を行う。

最低限の確認対象:

- 必須列の存在
- 数値列の数値変換可否
- `layout_name` の空値
- `map_item_id` の空値
- 行単位の読み込み失敗をログへ記録できること

### 4.9 行単位追跡

エラーや警告の特定を容易にするため、必要に応じて内部データ構造は元 CSV 行番号を保持してよい。

### 4.10 QGIS API 依存の分離

CSV 読込段階の内部データに、直接 `QgsLayoutItemMap` や `QgsRectangle` を埋め込むことを前提にしない。

QGIS オブジェクトへの変換は適用処理の段階で行う。

### 4.11 結果オブジェクト

主要処理は `True` / `False` だけを返さず、結果オブジェクトを返すこと。

対象処理は少なくとも次を含む。

- CSV エクスポート
- CSV インポート
- レイアウト再作成
- 地図コピー元スナップショット取得
- 地図アイテム任意コピー
- expression 生成
- expression 適用
- zvalue 再設定
- レイアウト一覧再読込
- CSV 読み込み
- 地図アイテム選択候補の再読込
- レイヤ一覧再読込
- Layout Designer 再描画

結果オブジェクトは少なくとも次を表現できること。

- 成功可否
- 致命的エラー有無
- 警告有無
- 処理対象件数
- 成功件数
- 警告付き件数
- 失敗件数
- ログメッセージ一覧
- レイアウト単位の結果
- 地図アイテム単位の警告や失敗内容

ログメッセージは少なくとも次の属性を持てること。

- レベル
- 表示文言
- 必要に応じて対象レイアウト名
- 必要に応じて対象地図アイテム ID
- 必要に応じて CSV 行番号

レイアウト単位の結果は少なくとも次を表現できること。

- レイアウト名
- 対象件数
- 反映成功件数
- 警告有無
- エラー有無
- `_要確認` 要否
- メッセージ一覧

地図アイテム単位の結果は必要に応じて次を保持してよい。

- `map_item_id`
- 処理成功可否
- 警告内容
- エラー内容

UI 層は結果オブジェクトから次を反映する。

- ログ表示領域への追記
- 処理完了後の概要表示
- 必要に応じた選択一覧の再読込
- `_要確認` 付与結果の表示

サービス層は UI 部品を直接更新してはならない。

## 5. UI 共通仕様

### 5.1 UI 構成方針

- UI は単一ダイアログ内のタブ切り替え構成とする
- 不要な追加ダイアログは表示しない
- 進捗表示専用ダイアログは表示しない
- 単なるメッセージ表示専用ダイアログは表示しない
- 可能な限りメインダイアログ内で操作と表示を完結させる

### 5.2 タブ構成

少なくとも次のタブを持つこと。

- エクスポート
- インポート
- 再作成
- 地図コピー
- expression ビルダ
- zvalue 再設定

### 5.3 共通ログ表示領域

ダイアログ下部に共通ログ表示領域を配置する。

- 表示対象は `INFO` `WARNING` `ERROR`
- スクロール可能とする
- 例:
  - `INFO: エクスポート開始`
  - `WARNING: 地図1がテンプレートで見つかりません`
  - `ERROR: CSV読み込み失敗`

### 5.4 共通レイアウト一覧表示

レイアウト選択は、用途に応じてチェックボックス付きリストまたはプルダウンで表示する。

チェックボックス付き複数選択リストには、同一タブ内で `全選択` と `全選択解除` を提供する。

ボタンは対象ラベルの直下、対象リストの直上に横並びで配置する。

一覧表示では少なくとも次を表示できること。

- レイアウト名
- 地図アイテム数

表示形式例:

`LayoutA (3)`

### 5.5 再読込ボタン

再読込ボタンは、何を再読込する操作かをボタン文言で明示すること。

例

- `レイアウト再読込`
- `CSV再読込`
- `レイアウト/レイヤ再読込`

### 5.6 メッセージ表示

警告、エラー、処理結果は追加ポップアップに過度に依存せず、メインダイアログ内で確認できる構成を優先する。

処理継続不能な致命的エラーについても、可能な限りメインダイアログ内で明示する。

## 6. 機能仕様

### 6.1 エクスポート

#### 6.1.1 対象

ユーザーはエクスポート対象レイアウトを複数選択できる。

#### 6.1.2 出力

出力先はユーザーが指定する単一の CSV ファイルとし、複数レイアウトの情報を 1 つの CSV に格納する。

#### 6.1.3 UI

少なくとも次を配置する。

- レイアウト一覧
- レイアウト一覧用 `全選択` ボタン
- レイアウト一覧用 `全選択解除` ボタン
- レイアウト再読込ボタン
- CSV 保存先入力欄
- 参照ボタン
- エクスポート実行ボタン

#### 6.1.4 操作

1. エクスポート対象レイアウトを選択する
2. CSV 保存先を指定する
3. エクスポートを実行する

#### 6.1.5 入力検証

次の場合は `ERROR` を表示する。

- レイアウト未選択
- CSV パス未指定

#### 6.1.6 ファイル入力欄表示

CSV 保存先入力欄には次が判別できる表示を持たせること。

- 出力用途であること
- 対象拡張子が `.csv` であること

### 6.2 インポート

#### 6.2.1 概要

CSV に保存された地図アイテム情報を既存レイアウトへ適用する。

新しいレイアウトは作成しない。

#### 6.2.2 入力

ユーザーは次を指定する。

- CSV ファイル
- 適用対象レイアウト

CSV に複数レイアウトが含まれる場合、実行対象は **プロジェクト側レイアウト一覧** で選択する。

CSV 内レイアウト一覧は参照表示として扱う。

#### 6.2.3 適用処理

CSV レコードごとに次を行う。

1. `layout_name` に一致する既存レイアウトを取得する
2. `map_item_id` を使用して `itemById()` を実行する
3. 一致した地図アイテムが見つかった場合、Extent と `dataDefinedMapLayers.expression` を設定する
4. 見つからない場合は警告を出力し、処理は継続する

#### 6.2.4 レイアウト未存在

CSV の `layout_name` に完全一致する既存レイアウトが存在しない場合は警告を出力し、他レイアウトの処理は継続する。

#### 6.2.5 実行時再読込

インポート実行時は、入力欄で指定された CSV を処理直前に再読込してから適用処理を開始する。

事前読込済みキャッシュのみを使って実行してはならない。

#### 6.2.6 UI

少なくとも次を配置する。

- レイアウト再読込ボタン
- CSV ファイル入力欄
- 参照ボタン
- CSV 内レイアウト一覧
- 適用対象レイアウト一覧
- 適用対象レイアウト一覧用 `全選択` ボタン
- 適用対象レイアウト一覧用 `全選択解除` ボタン
- インポート実行ボタン

CSV 読み込み時は `layout_name` とレイアウトごとの地図アイテム数を取得し、`レイアウト名 (件数)` 形式で一覧表示する。

`CSV 内レイアウト一覧` は参照表示のため、`全選択` と `全選択解除` の対象外とする。

#### 6.2.7 ファイル入力欄表示

CSV ファイル入力欄には次が判別できる表示を持たせること。

- 入力用途であること
- 対象拡張子が `.csv` であること
- 本プラグインのエクスポート結果 CSV を想定していること

### 6.3 再作成

#### 6.3.1 入力

ユーザーは次を指定する。

- CSV ファイル
- 新レイアウトテンプレート（`.qpt`）
- 再作成対象レイアウト

CSV に複数レイアウトが含まれる場合、再作成対象は CSV 内レイアウト一覧から選択可能とする。

#### 6.3.2 テンプレート

1 回の再作成処理につき 1 つのテンプレートを使用する。

#### 6.3.3 レイアウト名

CSV の `layout_name` をそのまま使用する。

同名レイアウトが存在する場合は、既存レイアウトを削除してから新レイアウトを作成する。

#### 6.3.4 処理

対象レイアウトごとに次を行う。

1. 新テンプレートからレイアウトを作成する
2. 対応する地図アイテムを `itemById()` で取得する
3. 見つかった場合、Extent と `expression` を設定する
4. 見つからない場合は警告を出力し、他アイテムの処理は継続する

#### 6.3.5 UI

少なくとも次を配置する。

- レイアウト再読込ボタン
- CSV ファイル入力欄
- 参照ボタン
- CSV 内レイアウト一覧
- CSV 内レイアウト一覧用 `全選択` ボタン
- CSV 内レイアウト一覧用 `全選択解除` ボタン
- テンプレート入力欄
- 参照ボタン
- 再作成実行ボタン

CSV 内レイアウト一覧は `レイアウト名 (件数)` 形式で表示し、再作成実行対象の主入力として扱う。

#### 6.3.6 実行時再読込

再作成実行時は CSV を処理直前に再読込してから処理を行う。

#### 6.3.7 ファイル入力欄表示

CSV 入力欄とテンプレート入力欄には次を明示すること。

- CSV 欄は入力 CSV（エクスポート結果）であり、対象拡張子が `.csv` であること
- テンプレート欄はレイアウトテンプレートであり、対象拡張子が `.qpt` であること

### 6.4 地図アイテム任意コピー

#### 6.4.1 概要

任意の元地図アイテムから取得した情報を、任意のコピー先地図アイテムへ適用する。

CSV 機能とは独立して利用できること。

#### 6.4.2 UI

少なくとも次を配置する。

- レイアウト再読込ボタン
- 元レイアウト選択プルダウン
- 元地図アイテム選択プルダウン
- 取得ボタン
- 取得値表示テキストボックス
- コピー先レイアウト選択プルダウン
- コピー先地図アイテム複数選択リスト
- コピー先地図アイテム一覧用 `全選択` ボタン
- コピー先地図アイテム一覧用 `全選択解除` ボタン
- コピー対象選択チェックボックス（`EXTENT` / `expression`）
- コピー実施ボタン

#### 6.4.3 取得

取得ボタン押下時に次を取得する。

- Extent（`xmin` `ymin` `xmax` `ymax`）
- `dataDefinedMapLayers.expression`

取得した値は、コピー実施時点での再取得ではなく、取得ボタン押下時点のスナップショットとして保持する。

#### 6.4.4 表示

取得値表示欄は読み取り専用とし、少なくとも次を表示する。

- 元レイアウト名
- 元地図アイテム識別表示
- Extent 4 値
- expression

#### 6.4.5 コピー対象選択

コピー時の適用対象は次から選択する。

- `EXTENT` のみ
- `expression` のみ
- `EXTENT` と `expression` の両方

いずれも未選択で実行した場合はエラーとする。

#### 6.4.6 コピー先選択

コピー先地図アイテムは複数選択可能とする。

同一 `map_item_id` が複数存在する場合は、表示上で連番を付与して個別選択可能にする。

例

- `map1 (#1)`
- `map1 (#2)`

内部的には表示名ではなく、`map_item_id` と `occurrence_index` を保持する。

#### 6.4.7 同一レイアウト許容

元レイアウトとコピー先レイアウトに同一レイアウトを選択してよい。

#### 6.4.8 スナップショット無効化

次の契機で保持中スナップショットを破棄し、再取得を必須とする。

- 元レイアウト変更時
- 元地図アイテム変更時

破棄後の取得値表示欄は、直前のスナップショット内容を残さず、未取得状態であることが判別できる表示へ直ちに戻すこと。

#### 6.4.9 エラー処理

少なくとも次を `ERROR` として扱う。

- 元レイアウト未選択
- 元地図未選択
- コピー先レイアウト未選択
- コピー先地図未選択
- コピー対象未選択
- 取得スナップショット未作成
- QGIS API 呼び出し失敗

### 6.5 expression ビルダ

#### 6.5.1 概要

プロジェクト内のレイヤ一覧からユーザーが複数レイヤを選択し、順位を指定して expression を組み立てる。

作成した expression は、任意の既存レイアウト内の任意の複数地図アイテムへ適用できる。

#### 6.5.2 UI

少なくとも次を配置する。

- レイアウト/レイヤ再読込ボタン
- 利用可能レイヤ一覧
- expression 対象レイヤ一覧
- 追加 / 除去ボタン
- 順位変更ボタン（上へ / 下へ）
- expression 生成ボタン
- 生成 expression 表示欄
- 適用先レイアウト選択
- 適用先地図アイテム複数選択リスト
- 適用先地図アイテム一覧用 `全選択` ボタン
- 適用先地図アイテム一覧用 `全選択解除` ボタン
- expression 適用ボタン

`利用可能レイヤ一覧` と `expression 対象レイヤ一覧` は、選択して追加 / 除去する操作を主とするため、`全選択` と `全選択解除` の対象外とする。

#### 6.5.3 レイヤ一覧

利用可能レイヤ一覧は、現在プロジェクトのレイヤを **レイヤパネル順** で表示する。

#### 6.5.4 expression 生成規則

expression は、expression 対象レイヤ一覧の上から順にレイヤ名を連結して作成する。

- 区切り文字は `|`
- 全体を `'` で括る
- レイヤ名に `'` が含まれる場合は `''` へエスケープする

例

レイヤ順が `Road`, `Label` の場合:

`'Road|Label'`

#### 6.5.5 適用範囲

初版では、1 回の適用処理で選択できる適用先レイアウトは単一レイアウトのみとする。

同一レイアウト内の地図アイテムは複数選択可能とする。

#### 6.5.6 適用処理

適用処理では次を行う。

1. 生成済み expression を取得する
2. 適用先レイアウトを取得する
3. 適用先地図アイテムを `map_item_id` と `occurrence_index` で解決する
4. 各地図アイテムへ `dataDefinedMapLayers.expression` を設定する

一致しない地図アイテムがある場合は警告を出し、他の地図アイテムへの適用は継続する。

#### 6.5.7 入力検証

次は `ERROR` として扱う。

- expression 対象レイヤ未選択
- 生成済み expression 未作成
- 適用先レイアウト未選択
- 適用先地図アイテム未選択
- QGIS API 呼び出し失敗

#### 6.5.8 生成 expression の無効化

次の操作が行われた場合、生成済み expression は未生成状態へ戻す。

- expression 対象レイヤへの追加
- expression 対象レイヤからの除去
- expression 対象レイヤの並び順変更
- レイヤ再読込で対象レイヤ一覧が変化した場合

### 6.6 ページ考慮 zvalue 再設定

#### 6.6.1 概要

レイアウト内の全レイアウトアイテム（ページアイテム除く）を対象に、ページを意識して zvalue を再設定する。

既存の CSV 系機能とは独立した手動実行機能とする。

#### 6.6.2 UI

少なくとも次を配置する。

- レイアウト再読込ボタン
- 処理対象レイアウト複数選択リスト
- 処理対象レイアウト一覧用 `全選択` ボタン
- 処理対象レイアウト一覧用 `全選択解除` ボタン
- zvalue 再設定実行ボタン

#### 6.6.3 対象

対象はレイアウト内の **全レイアウトアイテム** とする。

ただし `QgsLayoutItemPage` は対象外とし、ページアイテムの zvalue（0）は変更しない。

#### 6.6.4 ページ所属判定

各アイテムの所属ページは、アイテムのシーン矩形中心点で判定する。

- 中心点が含まれる最初のページを所属ページとする
- どのページにも属さないアイテムはページ外アイテムとして扱う

#### 6.6.5 再設定規則

ページ内での既存重なり順は維持する。

ページ間の優先規則は次の通り。

- 1 ページ目を最前面
- 2 ページ目以降はページ番号が大きいほど背面

前面ほど zvalue が大きい値になるよう連番で再設定する。

ページ外アイテムは警告を出力したうえで、最後尾として処理する。

#### 6.6.6 Layout Designer 反映

対象レイアウトの Layout Designer が開いている場合は、アイテムパネル表示へ更新を反映する。

- 開いている Designer 一覧から対象レイアウトを特定する
- レイアウト再描画により反映する
- Designer 未オープンは失敗扱いにしない
- Designer 再描画失敗は `WARNING` とし、zvalue 再設定自体は継続可能とする
- Designer 再描画は、zvalue 再設定処理本体が `success=True` の場合にのみ試行する

#### 6.6.7 エラー処理

少なくとも次をログへ反映する。

- `INFO`: 処理完了、対象なし
- `WARNING`: ページ外アイテム検出、Designer 再描画失敗
- `ERROR`: レイアウト未存在、QGIS API 呼び出し失敗

複数レイアウト選択時は、1 件失敗しても他レイアウト処理を継続する。

## 7. 実装制約

### 7.1 UI とロジックの分離

UI 定義と処理ロジックは分離すること。

次を同一ファイルに密結合させない。

- UI レイアウト定義
- イベント接続
- 業務ロジック
- QGIS 操作
- CSV 入出力処理

### 7.2 UI ファイル形式

- UI は Qt Designer で修正可能な `.ui` ファイルとして管理する
- 少なくともメインダイアログの UI は `.ui` ファイルとして保持する
- `.py` のみで UI を直接組み立てる実装を主方式にしない

### 7.3 責務分割

- UI、内部モデル、CSV 入出力、QGIS レイアウト操作、ログ表示の責務を分離する
- 単一ファイルに大量の処理を集約しない
- プラグイン起動コードには、アクション登録とダイアログ起動以上の業務ロジックを書きすぎない
- UI 層は `.ui` 読み込み、入力取得、表示更新に集中する
- CSV 処理はサービス層へ分離する
- レイアウト適用と再作成処理はサービス層へ分離する
- 内部データ構造はモデル層として分離する
- モデル層は可能な限り QGIS API に依存させない

### 7.4 想定構成

固定ではないが、少なくとも次の責務分離を満たすこと。

- `plugin.py` などのエントリポイント層
- `ui/` などの UI 定義
- `dialogs/` などの UI 制御層
- `models/` などの内部データ構造
- `services/` などの業務処理
- `utils/` などの補助処理

### 7.5 エントリポイント層

エントリポイント層は、QGIS からの起動、アクション登録、メインダイアログ表示を担当する。

CSV 解析、レイアウト反映、再作成ロジックを直接大量に書かないこと。

### 7.6 UI 層

UI 層は次の責務のみを持つこと。

- `.ui` の読み込み
- ウィジェット初期化
- シグナル接続
- 入力値の取得
- 実行結果の表示
- ログ表示欄へのメッセージ反映
- UI 契機での再読込

UI 層に CSV の詳細な解析や、QGIS API を用いた重い業務ロジックを集中させないこと。

### 7.7 サービス層

サービス層は業務処理を担当する。

少なくとも次は独立した責務として分離すること。

- CSV 読み込み
- CSV 書き出し
- レイアウト情報の抽出
- 既存レイアウトへの適用
- 新テンプレートからのレイアウト再作成

サービス層は UI ウィジェットを直接操作しないこと。

### 7.8 依存方向

依存方向は原則として次を守る。

- エントリポイント層 → UI 層
- UI 層 → サービス層 / モデル層
- サービス層 → モデル層 / ユーティリティ層
- モデル層 → 他層へ依存しない

### 7.9 UI 層の直接依存禁止

UI 層は `services` と `models` を介して業務処理へアクセスすること。

次を禁止する。

- `dialogs/*.py` から `utils/qgis_layout_helpers.py` への直接 import
- UI 層での `QgsProject` `layoutManager` `openLayoutDesigners` などの QGIS API 直接呼び出し

### 7.10 QGIS API 利用箇所

QGIS API を使う処理は、可能な限りサービス層または専用ヘルパーへ閉じ込める。

次の参照系処理は UI 層で直接実装せず、サービス層へ分離すること。

- レイアウト一覧再読込
- レイアウト単位の地図アイテム選択候補取得
- レイヤパネル順レイヤ一覧取得
- Layout Designer 再描画処理

上記サービスは UI へ返すデータと `OperationResult` を返却すること。

### 7.11 テスト容易性

CSV 解析、内部データ変換、警告判定など、QGIS GUI に依存しない処理は将来的に単体テストしやすい構成を意識すること。

少なくとも次は UI から分離する。

- CSV の読み書き
- 行バリデーション
- レイアウト名ごとのグルーピング
- warning / error 判定ロジック

### 7.12 例外処理

少なくとも次の失敗を防御し、失敗時は UI にエラー表示すること。

- ファイル I/O
- CSV 読み込み
- テンプレート読み込み
- レイアウト作成
- レイアウト上書き時の既存レイアウト削除
- `itemById()` による地図アイテム取得失敗
- QGIS API 呼び出し失敗
