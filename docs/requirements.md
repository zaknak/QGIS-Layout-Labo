# QGIS Layout Map Transfer Plugin - Requirements

## 1. 概要

本プラグインは、既存のQGISレイアウトから地図アイテムの状態をエクスポートし、既存レイアウトへの再適用、または新しいレイアウトテンプレートを使用したレイアウト再作成を行うためのツールである。

レイアウトテンプレートが変更された場合でも、既存レイアウトで調整済みの以下の情報を維持したまま、新テンプレートを用いてレイアウトを再生成できる。

対象情報

- 地図アイテムの表示範囲（Extent）
- 地図アイテムの data-defined map layers expression

---

## 2. 動作環境

- QGIS **3.4以上**
- Python（PyQGIS）

---

## 3. 用語定義

### 3.1 地図アイテムの表示内容

本プラグインにおける「表示内容」とは、QGISレイアウトの地図アイテムに設定された以下の data-defined property を指す。

`dataDefinedMapLayers.expression`

レイアウトXMLでは次の位置に存在する。

`LayoutItem > dataDefinedProperties > properties > dataDefinedMapLayers > expression`

例

`<Option type="QString" value="'Google Road'" name="expression"/>`

以下の設定は本プラグインの対象外とする。

- レイヤ可視状態
- 地図テーマ
- 縮尺
- 回転
- ラベル設定
- その他描画設定

---

## 4. 機能概要

本プラグインは次の3つの機能を提供する。

### 4.1 レイアウト情報エクスポート

既存レイアウトから地図アイテム情報を取得しCSVへ出力する。

### 4.2 レイアウト情報インポート

CSVの情報を使用し、既存レイアウトへ地図アイテム情報を適用する。

### 4.3 レイアウト再作成

CSVの情報を使用し、新しいレイアウトテンプレートからレイアウトを再生成する。

---

## 5. エクスポート機能

### 5.1 対象レイアウト

ユーザーはエクスポート対象のレイアウトを選択できる。

複数選択可能。

### 5.2 出力ファイル

出力先はユーザーが指定する **単一のCSVファイル** とする。

複数レイアウトの情報を1つのCSVに格納する。

### 5.3 CSV仕様

#### エンコーディング

BOM付き UTF-8

#### 文字列

すべて `"` で括る。

#### カラム構成

| 列 | 内容 |
|---|---|
| `layout_name` | レイアウト名 |
| `map_item_id` | 地図アイテムID |
| `xmin` | Extent xmin |
| `ymin` | Extent ymin |
| `xmax` | Extent xmax |
| `ymax` | Extent ymax |
| `expression` | `dataDefinedMapLayers.expression` |

### 5.4 Extentの扱い

Extentは次の4列で出力する。

- `xmin`
- `ymin`
- `xmax`
- `ymax`

数値は **不要な丸めを行わず float を文字列化する。**

小数点以下桁数の固定は行わない。

---

## 6. レイアウト再作成機能

### 6.1 入力

ユーザーは以下を指定する。

- CSVファイル
- 新レイアウトテンプレート（`.qpt`）
- 再作成対象レイアウト

CSVに複数レイアウトが含まれる場合、再作成対象は選択可能とする。

### 6.2 テンプレート

1回の再作成処理につき **1つのテンプレート** を使用する。

---

## 7. 地図アイテム対応

### 7.1 対応判定

地図アイテムの対応は `itemById()` により行う。

対応の起点は **CSVデータ側の地図アイテム** とする。

### 7.2 ID仕様

IDは必ずしもユニークではない。

`itemById()` が返す **最初の一致アイテム** を対象とする。

### 7.3 ID空文字

地図アイテムIDが空文字の状態は想定しない。

この状態の挙動は保証しない。

---

## 8. 再作成処理

CSVレコードごとに次の処理を行う。

1. 新テンプレートからレイアウトを作成する
2. 対応する地図アイテムを `itemById()` で取得する
3. 見つかった場合
   - Extent を設定する
   - `expression` を設定する
4. 見つからない場合
   - 警告を出力する
   - 一致した他アイテムの処理は継続する

---

## 9. 地図アイテム数の不一致

以下の場合は警告する。

- テンプレートの地図アイテム数 < CSVデータの地図アイテム数
- テンプレートの地図アイテム数 > CSVデータの地図アイテム数

比較は **レイアウト内の全地図アイテム数** で行う。

### 9.1 テンプレート側が少ない場合

移せる地図アイテムのみ反映する。

### 9.2 テンプレート側が多い場合

余った地図アイテムは **テンプレート初期値のまま残す**。

---

## 10. レイアウト名

CSVの `layout_name` をそのまま使用する。

同名レイアウトが存在する場合は、既存レイアウトを削除してから新レイアウトを作成する。

警告が発生した場合は、レイアウト名末尾に `_要確認` を付与する。

例

`市街図_要確認`

---

## 11. 警告

警告が発生した場合は、次の両方を行う。

- メインダイアログ内へメッセージ表示
- 再作成レイアウト名末尾への `_要確認` 付与

---

## 12. 警告内容

警告は次の単位で出力する。

### 12.1 レイアウト単位

例

`レイアウトAで不一致が発生しました`

### 12.2 地図アイテム単位

例

`レイアウトA: 地図1 がテンプレートで見つかりません`

## 13. UI

本プラグインのUIは **単一ダイアログ内のタブ切り替え式** とする。

機能ごとにタブを分ける。

想定タブ

- エクスポート
- インポート
- 再作成

### 13.1 UI構成方針

- 不要な追加ダイアログは表示しない
- 進捗表示専用ダイアログは表示しない
- 単なるメッセージ表示専用ダイアログは表示しない
- 可能な限りメインダイアログ内で操作・表示を完結させる

### 13.2 レイアウト一覧の再読込

UI操作を契機として、プロジェクト内のレイアウト状態を適切に読み込み直すこと。

少なくとも以下のタイミングでは、レイアウト一覧や関連状態を再取得する。

- ダイアログ初回表示時
- エクスポートタブ表示時
- インポートタブ表示時
- 再作成タブ表示時
- ユーザーが明示的に再読込を要求した時

警告・エラー・処理結果は、追加ポップアップに過度に依存せず、メインダイアログ内で確認できる構成を優先する。

処理継続不能な致命的エラーについても、可能な限りメインダイアログ内で明示する。

## 14. レイアウト情報インポート

CSVに保存された地図アイテム情報を既存レイアウトへ適用する。

本機能は新しいレイアウトを作成しない。

### 14.1 入力

ユーザーは以下を指定する。

- CSVファイル
- 適用対象レイアウト

CSVに複数レイアウトが含まれる場合、適用対象は選択可能とする。

### 14.2 レイアウト対応

CSVの `layout_name` とプロジェクト内レイアウト名が **完全一致** したものだけを適用対象とする。

レイアウト名の部分一致、あいまい一致、手動対応付けは行わない。

### 14.3 適用処理

CSVレコードごとに次の処理を行う。

1. `layout_name` に一致する既存レイアウトを取得する
2. `map_item_id` を使用して `itemById()` を実行する
3. 一致した地図アイテムが見つかった場合
   - Extent を設定する
   - `dataDefinedMapLayers.expression` を設定する
4. 見つからない場合
   - 警告を出力する
   - 処理は継続する

### 14.4 レイアウト未存在

CSVの `layout_name` に完全一致する既存レイアウトが存在しない場合

- 警告を出力する
- 処理は継続する

新しいレイアウトは作成しない。

### 14.5 地図アイテム数の不一致

CSV内レイアウトと既存レイアウトの全地図アイテム数が一致しない場合は警告する。

比較は **全地図アイテム数** で行う。

一致した地図アイテムがある場合は、そのアイテムへの適用は継続する。

## 22. 内部データ構造仕様

本プラグインは、CSV入出力とQGISレイアウト操作の間に内部データ構造を持つこと。

CSVを直接その場でQGISレイアウトへ反映する密結合実装は避ける。

内部データ構造は、少なくとも次の責務を分離できること。

- CSVの1行を表現する責務
- 1レイアウト分の地図アイテム群を表現する責務
- 複数レイアウト分のCSV全体を表現する責務

### 22.1 CSV1行相当のデータ

CSVの1行は、少なくとも次の情報を保持する構造として扱うこと。

- `layout_name`
- `map_item_id`
- `xmin`
- `ymin`
- `xmax`
- `ymax`
- `expression`

この構造は、1つの地図アイテムに対応する。

### 22.2 地図アイテムデータ

内部実装では、CSV1行相当の情報を「地図アイテムデータ」として扱うこと。

地図アイテムデータは、少なくとも次の意味を持つ。

- どのレイアウトに属するか
- どの地図アイテムIDに対応するか
- 適用すべき Extent
- 適用すべき `dataDefinedMapLayers.expression`

Extentは内部的には4つの数値として保持すること。

- `xmin`
- `ymin`
- `xmax`
- `ymax`

文字列1本に再結合した状態のみで保持してはならない。

### 22.3 レイアウト単位データ

同じ `layout_name` を持つ複数の地図アイテムデータを束ねて、1レイアウト分のデータとして扱うこと。

レイアウト単位データは、少なくとも次を保持できること。

- `layout_name`
- 地図アイテムデータの一覧

必要に応じて、処理中の補助情報を保持してよい。

例

- 警告有無
- 警告メッセージ一覧
- 元CSV行番号一覧

ただし、CSVの原始データ文字列に過度に依存しないこと。

### 22.4 CSV全体データ

CSVファイル全体は、複数のレイアウト単位データを保持する構造として扱うこと。

少なくとも次の操作を提供できること。

- `layout_name` 一覧の取得
- 特定 `layout_name` の取得
- レイアウト名単位での反復処理

### 22.5 数値型

`xmin` `ymin` `xmax` `ymax` は、内部的には数値として扱うこと。

CSV出力時のみ文字列化する。

CSV読み込み後に文字列のまま保持し続けないこと。

### 22.6 expression の扱い

`expression` は文字列として保持すること。

空文字は有効値として扱ってよい。

不要なトリム、加工、式変換は行わないこと。

CSVに記載された値をそのまま保持・適用すること。

### 22.7 レイアウト名と地図アイテムID

`layout_name` と `map_item_id` は文字列として保持すること。

比較時は requirements に定義された一致条件に従うこと。

- インポート時のレイアウト対応は `layout_name` の完全一致
- 地図アイテム対応は `itemById()` による一致

### 22.8 バリデーション責務

CSV読み込み時に、内部データ構造へ変換する前または変換時に最低限の妥当性確認を行うこと。

最低限の確認対象

- 必須列の存在
- 数値列の数値変換可否
- `layout_name` の空値
- `map_item_id` の空値を許容するかどうかは requirements の前提に従うこと
- 行単位の読み込み失敗時にログへ記録できること

本プラグインでは `map_item_id` 空値は想定外入力として扱ってよい。

### 22.9 行単位追跡

エラーや警告を特定しやすくするため、必要に応じて内部データ構造は元CSVの行番号を保持してよい。

保持する場合、ログや警告に利用してよい。

### 22.10 QGIS API 依存の分離

内部データ構造は、可能な限り QGIS API のオブジェクトに依存しないこと。

たとえば、CSV読込段階の内部データに直接 `QgsLayoutItemMap` や `QgsRectangle` を埋め込むことを前提にしない。

QGISオブジェクトへの変換は、適用処理の段階で行うこと。

## 23. ファイル構成と責務分割

本プラグインは、UI、CSV入出力、内部データ構造、QGISレイアウト操作、ログ表示の責務を分離した構成とすること。

単一ファイルに過度に処理を集中させないこと。

### 23.1 想定ディレクトリ構成

ディレクトリ構成は固定ではないが、少なくとも次の責務分離を満たすこと。

例

- `__init__.py`
- `metadata.txt`
- `plugin.py`
- `resources.py`
- `resources.qrc`
- `ui/`
  - `main_dialog.ui`
- `dialogs/`
  - `main_dialog.py`
- `models/`
  - `map_item_record.py`
  - `layout_record.py`
  - `csv_layout_dataset.py`
- `services/`
  - `csv_service.py`
  - `layout_export_service.py`
  - `layout_import_service.py`
  - `layout_rebuild_service.py`
- `utils/`
  - `logger.py`
  - `qgis_layout_helpers.py`
  - `csv_helpers.py`

上記と同等以上に責務分離されていれば、命名や分割単位は変更してよい。

### 23.2 エントリポイント層

プラグインのエントリポイント層は、QGISからの起動、アクション登録、メインダイアログ表示を担当する。

この層にCSV解析、レイアウト反映、再作成ロジックを直接大量に書かないこと。

### 23.3 UI層

UI層は、次の責務のみを持つこと。

- `.ui` の読み込み
- ウィジェット初期化
- シグナル接続
- 入力値の取得
- 実行結果の表示
- ログ表示欄へのメッセージ反映
- UI契機での再読込

UI層にCSVの詳細な解析ロジックや、QGIS APIを用いた重い業務ロジックを集中させないこと。

### 23.4 モデル層

モデル層は、内部データ構造を定義する責務を持つ。

少なくとも次を分離できること。

- CSV1行相当
- 1レイアウト分のデータ
- CSV全体データ

モデル層は、可能な限り QGIS API オブジェクトに依存しないこと。

### 23.5 サービス層

サービス層は、業務処理を担当する。

少なくとも次の処理は独立した責務として分離すること。

- CSV読み込み
- CSV書き出し
- レイアウト情報の抽出
- 既存レイアウトへの適用
- 新テンプレートからのレイアウト再作成

サービス層は、UIウィジェットを直接操作しないこと。

UIへの反映は、戻り値・結果オブジェクト・メッセージ一覧などを介して行うこと。

### 23.6 ユーティリティ層

共通的な補助処理はユーティリティ層に分離すること。

例

- ログ整形
- QGISレイアウト探索補助
- CSVフォーマット補助
- 数値変換補助

ただし、ユーティリティ層に業務処理全体を押し込めないこと。

### 23.7 依存方向

依存方向は原則として次を守ること。

- エントリポイント層 → UI層
- UI層 → サービス層 / モデル層
- サービス層 → モデル層 / ユーティリティ層
- モデル層 → 他層へ依存しない

UI層からモデル層を直接参照してよいが、モデル層からUI層へ依存してはならない。

### 23.8 QGIS API 利用箇所

QGIS API を使う処理は、可能な限りサービス層または専用ヘルパーへ閉じ込めること。

モデル層へ QGIS の重い依存を持ち込まないこと。

### 23.9 テスト容易性

CSV解析、内部データ変換、警告判定など、QGIS GUIに依存しない処理は、将来的に単体テストしやすい構成を意識すること。

少なくとも以下は UI から分離すること。

- CSVの読み書き
- 行バリデーション
- レイアウト名ごとのグルーピング
- warning / error 判定ロジック

## 24. 結果オブジェクト仕様

本プラグインの各主要処理は、処理結果を表す内部結果オブジェクトを返すこと。

UI層は、サービス層が返す結果オブジェクトをもとに、ログ表示・警告表示・処理結果表示を行うこと。

サービス層がUIウィジェットを直接更新してはならない。

### 24.1 対象処理

少なくとも次の処理は結果オブジェクトを返すこと。

- CSVエクスポート
- CSVインポート
- レイアウト再作成
- レイアウト一覧再読込
- CSV読み込み

### 24.2 結果オブジェクトの目的

結果オブジェクトは、少なくとも次を表現できること。

- 処理が成功したか
- 致命的エラーがあったか
- 警告があったか
- 処理対象件数
- 成功件数
- 警告付き件数
- 失敗件数
- ログメッセージ一覧
- レイアウト単位の結果
- 地図アイテム単位の警告や失敗内容

### 24.3 ログメッセージ

ログメッセージは、少なくとも次の属性を持てること。

- レベル
  - `INFO`
  - `WARNING`
  - `ERROR`
- 表示文言
- 必要に応じて対象レイアウト名
- 必要に応じて対象地図アイテムID
- 必要に応じてCSV行番号

ログメッセージは、UIのログ表示領域へそのまま表示できる形で保持してよい。

### 24.4 レイアウト単位の結果

各レイアウトについて、少なくとも次を表現できること。

- レイアウト名
- 対象件数
- 反映成功件数
- 警告有無
- エラー有無
- `_要確認` が必要かどうか
- メッセージ一覧

### 24.5 地図アイテム単位の結果

必要に応じて、各地図アイテムごとに次を保持してよい。

- `map_item_id`
- 処理成功可否
- 警告内容
- エラー内容

### 24.6 致命的エラーと継続可能エラー

結果オブジェクトは、少なくとも次を区別できること。

#### 致命的エラー

処理全体の続行が困難なもの。

例

- CSVファイルを開けない
- CSV必須列が不足している
- テンプレートを読めない

#### 継続可能エラー / 警告

一部不一致があっても処理継続可能なもの。

例

- `itemById()` で地図アイテムが見つからない
- レイアウト名が一致しない
- 地図アイテム数不一致

### 24.7 `_要確認` 判定

レイアウト単位の結果として、警告が発生したレイアウトは `_要確認` 判定を持てること。

再作成処理では、この判定に基づいてレイアウト名末尾へ `_要確認` を付与すること。

インポート処理では既存レイアウト名の変更は行わない。

### 24.8 件数集計

各主要処理の結果オブジェクトは、少なくとも次の集計値を保持できること。

- 処理対象レイアウト数
- 成功レイアウト数
- 警告付きレイアウト数
- 失敗レイアウト数

必要に応じて、地図アイテム件数も保持してよい。

### 24.9 UI反映

UI層は結果オブジェクトから次を反映すること。

- ログ表示領域への追記
- 処理完了後の概要表示
- 必要に応じた選択一覧の再読込
- `_要確認` 付与結果の表示

サービス層は、UI部品へ直接文字列を書き込まないこと。

## 25. CSV仕様

本プラグインで扱うCSVは、本プラグイン自身が出力し、本プラグイン自身が読み込むことを前提とした固定仕様とする。

一般的な任意CSVの読み込みを広くサポートすることは目的としない。

### 25.1 エンコーディング

CSVの文字コードは **BOM付きUTF-8** とする。

読み込み時は BOM付きUTF-8 を正として扱う。

### 25.2 ヘッダー

CSVはヘッダー行を必須とする。

ヘッダー名は固定とし、少なくとも次の列を持つこと。

- `layout_name`
- `map_item_id`
- `xmin`
- `ymin`
- `xmax`
- `ymax`
- `expression`

### 25.3 列順

CSVの列順は次の通り固定とする。

1. `layout_name`
2. `map_item_id`
3. `xmin`
4. `ymin`
5. `xmax`
6. `ymax`
7. `expression`

書き出し時は必ずこの順序で出力すること。

読み込み時も、この列順を前提とした固定仕様として扱ってよい。

ただし、実装上の負担が過大でなければ、ヘッダー名ベースで読み込んでもよい。

### 25.4 文字列列

文字列列は次を含む。

- `layout_name`
- `map_item_id`
- `expression`

これらはCSV出力時に `"` で括ること。

CSVライブラリの標準挙動により必要なエスケープが行われること。

### 25.5 数値列

数値列は次を含む。

- `xmin`
- `ymin`
- `xmax`
- `ymax`

内部的には数値として保持し、CSV出力時に文字列化する。

不要な丸めを行わない。

表示用の桁整形は行わない。

### 25.6 expression の保持

`expression` はCSVにそのまま保存すること。

不要な前処理・後処理を行わないこと。

以下を禁止する。

- 前後空白の自動除去
- クォート記法の書き換え
- 式の再解釈
- 式の正規化
- 任意のエスケープ置換を独自実装で行うこと

CSVとして必要なエスケープは、標準的なCSV書き出し処理に委ねること。

### 25.7 改行・引用符を含む文字列

`expression` や `layout_name` `map_item_id` に、CSV上でエスケープが必要な文字が含まれる可能性を考慮すること。

少なくとも次に対応できる実装とすること。

- カンマ
- `"`
- 改行

これらは独自実装で文字列連結してCSVを作らず、標準的なCSVライブラリを用いて入出力すること。

### 25.8 必須列不足

必須列が1つでも不足しているCSVは読み込み失敗とする。

この場合は **致命的エラー** として扱い、処理全体を中止する。

### 25.9 空値

#### `layout_name`

空値は不正とする。

該当行は読み込みエラーとして扱う。

#### `map_item_id`

空値は想定外入力とする。

該当行は読み込みエラーとして扱ってよい。

#### `xmin` `ymin` `xmax` `ymax`

空値は不正とする。

該当行は読み込みエラーとして扱う。

#### `expression`

空値を許容する。

### 25.10 数値変換失敗

`xmin` `ymin` `xmax` `ymax` の数値変換に失敗した行は不正行とする。

本プラグイン初版では **不正行が1行でもあればCSV全体を読み込み失敗とする。**

### 25.11 重複行

同一 `layout_name` かつ同一 `map_item_id` の行が複数存在する場合は不正CSVとみなしてよい。

本プラグイン初版では、同一 `layout_name` と同一 `map_item_id` の重複行を検出した場合、CSV読み込み失敗とする。

### 25.12 空ファイル

ヘッダーのみで実データ行が存在しないCSVは読み込み可能としてよいが、適用対象なしとして扱う。

この場合は処理継続可能な `WARNING` または `INFO` を出力する。

### 25.13 改行コード

改行コードは実行環境依存でよいが、少なくとも標準的なCSVライブラリを用いて互換性を確保すること。

独自の改行コード制御を主目的として実装を複雑化しないこと。

### 25.14 書き出しの一貫性

本プラグインが書き出したCSVは、本プラグイン自身でそのまま再読込可能であること。

エクスポート → インポート → 再作成 の流れで、CSVフォーマット差異により失敗しないことを優先する。

## 26. 地図アイテム任意コピー機能

任意の元地図アイテムから取得した情報を、任意のコピー先地図アイテムへ適用する機能を提供する。

本機能は既存のCSV機能とは独立して利用できること。

### 26.1 タブ構成

単一ダイアログ内に「地図コピー」タブを追加する。

同タブには少なくとも次を配置する。

- レイアウト再読込ボタン
- 元レイアウト選択プルダウン
- 元地図アイテム選択プルダウン
- 取得ボタン
- 取得値表示テキストボックス
- コピー先レイアウト選択プルダウン
- コピー先地図アイテム複数選択リスト
- コピー対象選択チェックボックス（`EXTENT` / `expression`）
- コピー実施ボタン

### 26.2 元地図からの取得

元地図選択後、取得ボタン押下時に次を取得する。

- Extent（`xmin` `ymin` `xmax` `ymax`）
- `dataDefinedMapLayers.expression`

取得した値は、コピー実施時点での再取得ではなく、取得ボタン押下時点のスナップショットとして保持する。

### 26.3 取得値表示

取得値表示テキストボックスには、少なくとも次を表示する。

- 元レイアウト名
- 元地図アイテム識別表示
- Extent 4値
- expression

表示テキストは読み取り専用とする。

### 26.4 コピー対象の選択

コピー実施時の適用対象はチェックボックスで選択する。

- `EXTENT` のみ
- `expression` のみ
- `EXTENT` と `expression` の両方

いずれも未選択の状態で実行した場合はエラーとする。

### 26.5 コピー先地図アイテムの複数選択

コピー先地図アイテムは複数選択可能とする。

同一 `map_item_id` が複数存在する場合は、表示上で連番を付与して個別選択可能にする。

例

- `map1 (#1)`
- `map1 (#2)`

内部的には表示名ではなく地図アイテム実体を識別できる情報を保持すること。

### 26.6 同一レイアウトの扱い

元レイアウトとコピー先レイアウトに同一レイアウトを選択することを許可する。

同一レイアウト内の別地図アイテムへのコピーを可能とする。

### 26.7 再読込契機

地図コピータブでも既存要件と同様に、レイアウト状態を固定キャッシュしないこと。

少なくとも次の契機で一覧や関連状態を再取得する。

- ダイアログ初回表示
- タブ切替時
- 地図コピータブの再読込ボタン押下時
- コピー実施後

### 26.8 エラー処理とログ

本機能は結果オブジェクトを返し、UIは結果オブジェクトを介して表示更新すること。

少なくとも次の異常系を `ERROR` として扱うこと。

- 元レイアウト未選択
- 元地図未選択
- コピー先レイアウト未選択
- コピー先地図未選択
- コピー対象チェック未選択
- 取得スナップショット未作成
- QGIS API 呼び出し失敗

## 27. expressionビルダ機能

プロジェクト内のレイヤ一覧からユーザーが複数レイヤを選択し、
expression 内でのレイヤ順位を指定して expression を組み立てる機能を提供する。

作成した expression は、任意の既存レイアウト内の任意の複数地図アイテムへ適用できる。

### 27.1 タブ構成

単一ダイアログ内に「expressionビルダ」タブを追加する。

少なくとも次を配置する。

- レイアウト/レイヤ再読込ボタン
- 利用可能レイヤ一覧（レイヤパネル順）
- expression対象レイヤ一覧
- 追加/除去ボタン
- 順位変更ボタン（上へ/下へ）
- expression生成ボタン
- 生成expression表示欄（読み取り専用）
- 適用先レイアウト選択
- 適用先地図アイテム複数選択リスト
- expression適用ボタン

### 27.2 レイヤ一覧

利用可能レイヤ一覧は、現在プロジェクトのレイヤを **レイヤパネル順** で表示する。

固定キャッシュは禁止し、少なくとも次の契機で再取得する。

- ダイアログ初回表示
- タブ切替時
- 再読込ボタン押下時
- expression適用処理完了後

### 27.3 expression生成規則

expressionは、expression対象レイヤ一覧の上から順にレイヤ名を連結して作成する。

生成規則

- 区切り文字: `|`
- 全体を `'` で括る

例

- レイヤ順が `Road`, `Label` の場合: `'Road|Label'`

レイヤ名に `'` が含まれる場合は、`''` へエスケープして生成する。

### 27.4 適用範囲

初版では、1回の適用処理で選択できる適用先レイアウトは **単一レイアウトのみ** とする。

同一レイアウト内の地図アイテムは複数選択可能とする。

### 27.5 適用処理

適用処理では次を行う。

1. 生成済みexpressionを取得する
2. 適用先レイアウトを取得する
3. 適用先地図アイテムを解決する
4. 各地図アイテムへ `dataDefinedMapLayers.expression` を設定する

地図アイテム解決は既存仕様と同様に `itemById()` 相当の一致ルールに従う。

一致しない地図アイテムがある場合は警告を出し、他の地図アイテムへの適用は継続する。

### 27.6 入力検証

次は `ERROR` として扱う。

- expression対象レイヤ未選択
- 生成済みexpression未作成
- 適用先レイアウト未選択
- 適用先地図アイテム未選択
- QGIS API 呼び出し失敗

### 27.7 結果オブジェクトとログ

本機能の主要処理（expression生成、expression適用）は、
既存ルールに従い結果オブジェクトを返すこと。

ログレベルは次を使い分ける。

- `INFO`: expression生成完了、適用完了
- `WARNING`: 一部地図アイテム不一致など継続可能異常
- `ERROR`: 入力不備、API失敗など処理続行不能異常
