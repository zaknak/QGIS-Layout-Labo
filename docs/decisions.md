# Layout Labo Decisions

## 2026-03-09 要件書の再構成方針

### 決定内容

`docs/requirements.md` は、追記履歴ではなく現在有効な仕様だけを持つ正本として管理する。

### 理由

- 実装時に参照すべき仕様を一意にしたい
- 同一テーマが複数箇所に分散すると、仕様衝突や見落としが起こりやすい
- 機能追加時に末尾追記を続けると、章構成が肥大化し保守性が落ちる

### 採用ルール

- 仕様変更時は `requirements.md` の既存該当節へ統合更新する
- 既存記述と重複または衝突する文言は更新または削除する
- 変更理由や検討経緯は `docs/decisions.md` へ記録する
- `requirements.md` へ文末追記で新章を増築する運用は採らない

### 影響範囲

- `docs/requirements.md` の章構成
- `AGENTS.md` の Codex 作業ルール

### requirements 反映箇所

- `docs/requirements.md` 1 章
- `AGENTS.md` 要件の参照節

## 2026-03-10 レビュー指摘に伴う要件補正

### 決定内容

`docs/requirements.md` の結果オブジェクト対象一覧に、地図コピー元スナップショット取得、地図アイテム任意コピー、expression 生成、expression 適用、zvalue 再設定を復帰する。

あわせて、地図コピー元の選択変更でスナップショットを破棄した際は、取得値表示欄も未取得状態へ戻すことを明記する。

### 理由

- 正本化後の要件書から一部処理の `OperationResult` 契約が落ちると、実装が機能ごとに不統一になりうる
- スナップショット破棄後も旧表示が残ると、内部状態と UI 表示が乖離し、誤操作を招く

### 影響範囲

- `docs/requirements.md` 4.11 結果オブジェクト
- `docs/requirements.md` 6.4.8 スナップショット無効化

## 2026-03-10 チェックボックス付きリストの一括選択UI追加

### 決定内容

各機能タブの `全選択` と `全選択解除` は、チェックボックスで処理対象を指定するリストに限定して追加する。

対象は、エクスポート、インポートの適用対象、再作成対象、地図コピーのコピー先地図、expression の適用先地図、zvalue 再設定対象の各リストとする。

`インポート > CSV内レイアウト` は参照表示のため対象外とし、`expression ビルダ > 利用可能レイヤ一覧 / expression 対象レイヤ一覧` は追加 / 除去操作を主とするため対象外とする。

ボタン配置は、対象ラベルの直下かつ対象リストの直上に横並びで統一する。

### 理由

- チェックボックス付きリストは処理対象の一括切替需要が高い
- 参照リストへ同じボタンを付けると、実行対象と誤認しやすい
- expression ビルダのレイヤ一覧は移動操作が主目的であり、チェック切替とは操作意味が異なる
- 配置位置を統一すると、タブごとに操作場所を探し直さずに済む

### 影響範囲

- `docs/requirements.md` 5.4 共通レイアウト一覧表示
- `docs/requirements.md` 6.1.3 エクスポート UI
- `docs/requirements.md` 6.2.6 インポート UI
- `docs/requirements.md` 6.3.5 再作成 UI
- `docs/requirements.md` 6.4.2 地図アイテム任意コピー UI
- `docs/requirements.md` 6.5.2 expression ビルダ UI
- `docs/requirements.md` 6.6.2 ページ考慮 zvalue 再設定 UI

## 2026-03-10 README を保守対象へ明文化

### 決定内容

`readme.md` を GitHub リポジトリの表紙兼簡易なユーザーマニュアルとして明示し、ユーザー向け機能、名称、操作手順、前提条件、UI 構成に変更がある場合は更新対象に含める。

更新責務は次のように分担する。

- `docs/requirements.md`: README の役割と更新条件を定義する
- `AGENTS.md`: 実装時に README 更新要否を確認する運用ルールを定義する

README は公開向け要約文書とし、詳細仕様の正本は引き続き `docs/requirements.md` とする。

### 理由

- README が古いままだと、初見ユーザーに現在の機能構成や操作方法を誤って案内してしまう
- 公開向け文書としての README と、実装判断用の正本仕様である requirements の役割を分けた方が保守しやすい
- `AGENTS.md` と `docs/requirements.md` の両方に明記すると、実装時の見落としと文書方針の抜けを同時に防げる

### 影響範囲

- `AGENTS.md` 要件の参照節
- `docs/requirements.md` 1. 文書方針
- `readme.md`
